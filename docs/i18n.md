# Shop-facing language (i18n)

The app renders its own chrome — page titles, nav, labels, buttons,
descriptive prose, toast/error messages — in **one language per
deployment**, chosen by the shop. This is *shop-facing*: a Taiwan shop
runs the whole app in Traditional Chinese, a Japanese shop in Japanese.
It is **not** per-user and **not** switchable at runtime. User-generated
content (event titles, notes) and legacy `EO_*` catalog data are never
translated.

## How selection works

The language is a build-time constant, read from the config seam exactly
like `locale.currency`:

```ts
// fundive.config.ts
locale: {
  timezone: 'Asia/Taipei',
  currency: 'TWD',
  currencyLabel: 'NTD',
  language: 'en',   // 'en' | 'zh-TW' | 'ja'
},
```

Because it's a constant, there is **no React context or provider**. The
catalog is resolved once at module load in `src/i18n/index.ts`:

```ts
import { t } from '../i18n'

<h1>{t.dashboard.title}</h1>
<span>{t.shell.pending(count)}</span>
```

`t` is a plain object — the message catalog for the configured language.

## The catalogs

```
src/i18n/
  index.ts              # resolves `t` from siteConfig.locale.language
  messages/
    en.ts               # source of truth; its type IS the contract
    zh-TW.ts            # const zhTW: Messages = { … }
    ja.ts               # const ja:   Messages = { … }
  messages.test.ts      # runtime shape-parity backstop
```

`en.ts` exports the catalog **and** its inferred type:
`export type Messages = typeof en`. Every other catalog is annotated
`const zhTW: Messages = { … }`, so a **missing or misnamed key fails the
TypeScript build** — English is the schema. `messages.test.ts` re-checks
the same shape at runtime and confirms the configured language resolves.

Rules for catalog files:

- **Pure data / plain functions only.** No imports from `src/config` or
  React. This keeps a catalog portable to the Deno edge-function runtime
  (see "Not yet covered" below).
- **No `as const`** on the catalog object — values must widen to `string`
  so a translation can supply a different literal for the same key.
- **Interpolation is a function.** A string that needs a runtime value is
  a function; its parameter names and types propagate to every
  translation through the shared `Messages` type:

  ```ts
  // en.ts
  greeting: (name: string) => `Hi ${name}`,
  // zh-TW.ts  — signature is enforced
  greeting: (name: string) => `${name} 您好`,
  ```

Catalogs are namespaced by area (`nav`, `common`, `shell`, `dashboard`,
`register`, `admin.*`, …). Keep a key where its text is used.

## Adding a string

1. Add it to `src/i18n/messages/en.ts` under the right namespace.
2. Add the **same key** to `zh-TW.ts` and `ja.ts` (the build fails until
   you do).
3. Replace the inline literal in the component with `t.namespace.key`.

Module-scope constants (e.g. a nav array) can reference `t` directly —
`t` is resolved at import time, before those constants initialize.

## Adding a language

1. Extend `SupportedLanguage` in `src/config/site.ts`.
2. Extend the zod enum in `src/config/site.schema.ts`.
3. Add `src/i18n/messages/<tag>.ts` (`const xx: Messages = { … }`) and
   register it in `src/i18n/index.ts`.
4. Bump `CONFIG_CONTRACT_VERSION` only if you also changed the config
   *shape* — adding an enum member does not require a bump.

> Here the deployment's `fundive.config.ts` reaches the app through the
> `virtual:fundive-config` module, typed as `SiteConfig`, so the literal
> is validated by zod (not the TypeScript literal check) and needs no
> `as const`. A deployment repo that imports the config file directly
> does add `as const` to narrow the literal to the union.

## Keeping the two repos in sync

The `src/i18n` module and all three catalogs are **identical** in
`app-fundivers` and the upstream `fundive` repo — only the config
*wiring* differs (direct import vs the `virtual:fundive-config` plugin).
Port catalog changes to both.

## Translation quality

Shipped translations are a strong first pass. **Dive terminology and a
shop's local register deserve a native review before that shop flips
`language` away from `en` in production.** The English build is
unaffected while translations are reviewed.

## Not yet covered (separate surfaces)

- **Emails** — done. The Deno edge functions read the same catalogs through
  `supabase/functions/_shared/i18n.ts`, a Deno-safe selector (the browser's
  `src/config/site.ts` path isn't resolvable there). One exception on purpose:
  the **partner-facing** half of a package recommendation stays English. It goes
  to a dive shop abroad, for whom your shop-facing language is meaningless. See
  `_shared/package-registration-email.ts`.
- **The registration PDF** — translated, but it needs a font. jsPDF's built-in
  helvetica is WinAnsi: it does not fail on CJK, it silently emits mangled bytes.
  `supabase/functions/_shared/pdf-cjk.ttf` (Noto Sans TC) is embedded and used
  per-string. A shop with Japanese divers must replace that file with Noto Sans
  JP — the two are not supersets of each other. `pdf-font-coverage.test.ts` fails
  with the missing glyphs named if you forget.
- **Legal text** — not translated, and not in code. Terms of Use and waivers are
  shop-authored in the admin UI (**Manage → Terms of Use** / **Waivers**), so you
  write them in your own language. The starter template is English-only on
  purpose: a machine translation of a legal document is not the same document,
  and shipping three "equivalent" versions invites the question of which one a
  diver actually agreed to.
